# 原理说明

本文说明当前版本脚本的实现逻辑。当前实现已经和早期版本不同，核心变化是：

- 学习流程固定走 **老界面**
- 评论与作业拆成 **独立任务窗口**
- 题型支持扩展到 **填空题 / 问答题 / 论述题**

---

## 1. 总体架构

当前整体分成两个并行部分：

### 1.1 学习窗口

负责：

- 打开老界面学习页 `home/learn/index#/课程ID/go`
- 自动播放视频
- 自动处理视频内弹题
- 自动处理测验

### 1.2 评论 / 作业窗口

负责：

- 按课程顺序进入成绩页 `/result`
- 读取作业与讨论进度
- 进入 `/homework` 处理未提交作业
- 进入讨论区补评论

这个窗口和学习窗口使用同一个 `BrowserContext`，因此：

- 登录只做一次
- Cookies / session 自动共享

---

## 2. 登录与验证码

浏览器通过 Playwright 启动真实 Chrome，并做了基础反自动化伪装：

- 隐藏 `navigator.webdriver`
- 伪造 `plugins`
- 伪造 `languages`
- 伪造 `window.chrome`

登录流程：

1. 打开首页
2. 点击登录按钮
3. 在 iframe 中定位账号密码框
4. 处理阿里云验证码
5. 提交登录

验证码处理方式不是直接调接口，而是：

- 在主页面和 iframe 中查找验证码元素
- 用鼠标轨迹模拟点击
- 检查登录按钮是否解锁

---

## 3. 课程收集与调度

登录后进入课程中心：

- `http://www.uooc.net.cn/home#/center/course/learn`

通过课程卡片上的：

- `继续学习`
- `开始学习`
- `查看课程`

提取课程 ID。

然后：

- 学习窗口按 `COURSE_CONCURRENCY` 并发学习
- 评论 / 作业窗口按课程顺序逐门处理

---

## 4. 为什么固定走老界面

新界面并不是所有课程都稳定可用。部分课程：

- 深链不稳定
- 页面结构差异大
- 某些课程甚至直接跳错误页

所以当前学习流程统一使用：

```text
http://www.uooc.net.cn/home/learn/index#/课程ID/go
```

老界面左侧目录使用 `.basic`、`.oneline`，结构更稳定，便于做：

- 当前节识别
- 下一节跳转
- 测验资源点击
- 讨论任务识别

---

## 5. 视频学习逻辑

视频部分由 `VideoTracker` 负责。

### 5.1 未看视频判定

脚本会记录每门课已看过的视频 URL 到：

```text
data/<username>/<courseId>.txt
```

再次运行时通过 URL 标准化后的值去重。

### 5.2 播放策略

在页面上下文中直接控制 `<video>`：

- 静音
- 倍速
- 自动继续播放

### 5.3 完成判定

任一条件成立即视为完成：

- `video.ended === true`
- 播放进度超过约 98.5%
- 剩余时间小于约 3 秒

### 5.4 视频内弹题

视频内弹题一般出现在 `#quizLayer`。

当前策略是穷举选项组合：

- 单选 / 多选都统一处理
- 提交后如果弹层消失即视为成功

---

## 6. 测验与作业的做题逻辑

### 6.1 题目来源

章节测验和作业最终都落在试卷页，题目容器通常是：

```text
.queContainer
```

### 6.2 当前支持的题型

根据平台前端脚本 `student_do_paper.js`，当前已经明确支持：

| 题型编码 | 含义 |
|---|---|
| `10` | 单选题 |
| `11` | 多选题 |
| `20` | 判断题 / 单选型任务 |
| `30` | 填空题 |
| `31` | 名词解释 |
| `40` | 问答题 |
| `50` | 组合阅读题 |
| `60` | 论述题 |

其中：

- `10 / 11 / 20` 走选项识别
- `30 / 31 / 40 / 60` 走文本答案识别与填写

### 6.3 客观题识别

客观题流程：

1. 对单题截图
2. 发给大模型
3. 要求只返回：

```json
{"answers":["A","C"]}
```

4. 在页面中点击对应选项

### 6.4 填空题 / 主观题识别

文本题流程：

1. 对单题截图
2. 发给大模型
3. 要求返回：

```json
{"answers":["答案1","答案2"]}
```

4. 将答案写入：
   - 普通 `input`
   - `textarea`
   - UEditor

这一步修复了旧版本“AI 有答案但页面没填进去就提交”的漏洞。

### 6.5 UEditor 处理

平台主观题常用 UEditor。

当前写入方式：

- 如果题目有 UEditor 容器 ID
- 通过 `window.UE.getEditor(id)` 获取编辑器实例
- 在 `ready()` 后调用 `setContent()`

同时也会回写普通输入节点，尽量兼容不同渲染方式。

---

## 7. 测验提交流程

提交流程：

1. 点击 `提交试卷`
2. 处理可能出现的验证码
3. 点击最终确认按钮
4. 检查是否出现“请重新提交测验”等失败提示

如果第一次未通过：

- 清空已有答案
- 切换重试模型
- 再做一轮

如果仍未通过，并且是客观题：

- 进入暴力组合兜底逻辑

---

## 8. 评论逻辑

评论逻辑已经从学习主流程中拆出，默认由独立任务窗口处理。

### 8.1 评论数据来源

使用以下接口：

- `Home/Threads/list`
- `Home/Threads/details`
- `Home/Threads/reply`

### 8.2 评论范围

当前会扫描三类讨论区：

- `type=10` 学生提问
- `type=20` 随堂讨论
- `type=30` 综合讨论

### 8.3 评论策略

不是在同一帖子里反复评论，而是：

1. 扫描新帖子
2. 跳过已经评论过的帖子
3. 读取帖子详情
4. 复制当前帖中已有的其他用户评论
5. 作为自己的回复发出

### 8.4 状态记录

每门课会记录：

```text
data/<username>/discussion_<courseId>.json
```

其中包含：

- 已评论过的帖子 ID
- 已使用过的评论内容
- 本地评论记录

### 8.5 限流处理

评论接口存在统一节流。

实测发现：

- 不同帖子之间共享限流
- 不同讨论区之间也共享限流

所以脚本实现了**全局评论队列**，默认冷却：

```text
DISCUSSION_INTERVAL_MS=65000
```

---

## 9. 评论 / 作业独立窗口工作流

独立窗口按课程顺序处理，每门课流程如下：

1. 打开 `/result`
2. 读取 `/home/course/progress?cid=...`
3. 计算：
   - 作业完成度
   - 讨论得分
4. 进入 `/homework`
5. 先做未提交作业
6. 再判断讨论是否需要补足
7. 进入讨论区补评论
8. 切下一门课程

### 9.1 作业优先级

当前优先级是：

```text
作业 > 评论
```

### 9.2 作业顺序

未提交作业会按以下顺序排序：

1. `end_time` 升序
2. `start_time` 升序
3. `id` 升序

即：

> 最早截止的作业优先做

### 9.3 讨论是否还要继续补

平台不会直接返回“还差多少条讨论”，所以当前根据成绩页返回的数据判断：

- `discuz`：讨论项权重
- `discuss_score`：当前讨论得分

当：

```text
discuss_score < discuz
```

时，视为讨论项未满分，继续补评论。

## 10. 启动控制逻辑

启动只保留两个问题：

```text
是否自动学习视频/测验 (Y/n):
是否评论/作业 (Y/n):
```

如果第二项为 `Y`，则自动启用：

- 评论
- 作业
- 评论 / 作业独立窗口

这样用户不需要再手动理解多个细分开关。

---

## 11. 模型调用方式

当前统一通过 OpenAI 兼容接口调用大模型。

配置项：

- `MODEL`
- `RETRY_MODEL`
- `BASE_URL`
- `API_KEY`

### 11.1 识图题

对截图转 base64 后以 `image_url` 方式发给模型。

### 11.2 文本题

同样基于题目截图，让模型直接返回文本答案数组。

### 11.3 当前问题定位

如果后续再次出现“识别失败”，通常需要优先排查：

1. API Key / 余额 / 配额
2. 截图是否完整
3. 题目是否主要由图片组成
4. 页面题型是否被正确识别
5. 文本答案是否真正写入了输入控件

---

## 12. 已知限制

- 平台 DOM 结构变化会导致定位失效
- 老界面深链不是所有课程都稳定，最好从课程主页或 `#/课程ID/go` 进入
- 题目全为图片时，截图质量会直接影响识别结果
- 讨论得分规则平台未公开精确公式，当前只能按得分是否满权重来判断是否继续补评论
- 主观题答案质量完全依赖模型
- 考试流程仍未纳入自动化范围
